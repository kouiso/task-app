# /review-pr - PRレビューチェックリスト

**ユーザーからPRレビューを依頼された際、以下の項目を必ず自動でチェックすること。**

## 1. 要件・仕様の整合性チェック

### Issue/資料との整合性確認

1. **関連Issue・資料の徹底調査**
   - PRに紐づいているIssueを`gh pr view`で確認
   - Issue内の要件・仕様・受け入れ条件を全て抽出
   - 関連する設計ドキュメント（`doc/`配下、`*.md`等）を検索・確認
   - Figmaリンク等の外部資料があれば内容を確認

2. **要件充足チェック**
   - Issueに記載された要件が**全て**実装されているか
   - 「〜すべき」「〜が必要」等の記述を見落としていないか
   - 受け入れ条件（Acceptance Criteria）を満たしているか

3. **仕様との相違チェック**
   - 設計ドキュメントと実装に齟齬がないか
   - API仕様（リクエスト/レスポンス形式、エラーハンドリング）が一致しているか
   - DB設計（テーブル名、カラム名、型）が一致しているか

## 2. コード品質・アンチパターンチェック

### アンチパターン検出

以下のアンチパターンを発見したら**必ず指摘**：

1. **TypeScript/JavaScript**
   - `any`型の使用（型安全性の放棄）
   - `as`による型アサーション（型ユーティリティで解決すべき）
   - ネストが深すぎる条件分岐（早期リターンで解決）
   - マジックナンバー・マジックストリング（定数化すべき）
   - 同じ処理の重複（共通化・抽象化すべき）

2. **React/React Native**
   - 不要な再レンダリングを引き起こす実装
   - useEffect内での依存配列の不備
   - 状態管理の不適切な設計（props drilling等）
   - インラインスタイルの乱用

3. **バックエンド（NestJS/Prisma）**
   - N+1クエリ（`include`や`select`で解決）
   - トランザクション漏れ（複数DB操作の整合性）
   - エラーハンドリングの不備（握り潰し等）
   - 環境変数の直接参照（DIやConfigServiceを使うべき）

4. **セキュリティ**
   - 機密情報のハードコーディング
   - 入力値のバリデーション不足
   - SQLインジェクション・XSSの脆弱性
   - 認証・認可チェックの漏れ

### より良い実装の提案

1. **可読性向上**: 命名改善、処理分割、早期リターン
2. **パフォーマンス向上**: 不要計算削除、メモ化、クエリ最適化
3. **保守性向上**: テスト容易性、拡張性、既存パターン準拠

## 3. コメント・差分の品質チェック

### コメント粒度チェック

PRの差分に含まれるコメントを全てチェックし、以下に該当するものは**削除を提案**：

1. **自明なコメント（What）** - 関数名・変数名で自明な処理の説明
2. **削除対象の判断基準** - 「このコメントを消したら、コードの意図が分からなくなるか？」 → NOなら削除
3. **残すべきコメント** - 非自明な意図・理由（Why）の説明、外部要因による制約の説明、警告・注意事項

### 不必要な差分チェック

1. **スコープ外の変更** - Issue/PRの目的と関係ない変更
2. **余計なフォーマット変更** - 目的と無関係なインデント変更
3. **不要なコードの追加** - 使われていないimport、変数、console.log

## 4. レビュー実行プロトコル

1. **関連資料の収集**: PRに紐づくIssue、設計ドキュメント、仕様書を全て確認
2. **要件リストの作成**: Issue/資料から要件を抽出し、チェックリスト化
3. **差分全体を確認**: `gh pr diff` または GitHub MCPで差分取得
4. **要件充足確認**: 各要件が実装されているか1つずつ確認
5. **コード品質チェック**: アンチパターン、改善点を洗い出し
6. **コメントスキャン**: 差分内の全コメントを品質基準に照らし合わせ
7. **問題点のリスト化**: 発見した問題は具体的な行番号と理由を添えて報告
8. **改善提案**: より良い実装方法があれば具体的なコードと共に提案

## 5. レビュー報告フォーマット

```
## レビュー結果

### ✅ LGTM / ⚠️ 要修正

### 要件充足状況
- [x] {要件1}: 実装済み
- [ ] {要件2}: 未実装または不完全（{理由}）

### 問題点（要修正の場合）

#### 要件・仕様関連
- [ ] {Issue/資料との相違点}

#### アンチパターン・コード品質
- [ ] `{ファイルパス}:{行番号}`: {問題の説明} → {改善案}

#### コメント関連
- [ ] `{ファイルパス}:{行番号}`: `{問題のコメント}` → 自明なため削除推奨

#### 差分関連
- [ ] `{ファイルパス}`: スコープ外の変更（{理由}）

### 改善提案（任意）
- {より良い実装方法の提案}

### 良い点
- {設計・実装で良かった点}
```

## 心得

- **要件は全て満たす** → 1つでも漏れがあれば指摘
- **仕様との整合性は絶対** → ドキュメントと実装の乖離は許さない
- **アンチパターンは見逃さない** → 将来の技術的負債を防ぐ
- **改善提案は具体的に** → 「こうした方がいい」だけでなくコード例も示す
- **レビューは徹底的に** → 見逃しはユーザーの負担になる
- **コメントは厳しくチェック** → 「What」コメントは即座に指摘
- **スコープは死守** → 「ついでに」は絶対に見逃さない

教材執筆・レビューの完全手順。書く前の思考矯正（guide 04 の6手順）→ 文章衛生ゲート(textlint) → 品質ゲート(check_quality.sh) → 初心者シミュレーションの4段構成。

## 書く前に必ず通す思考プロセス（guide 04 の6手順）

<!-- Source: hana652/tech-writing-pack guide 04 — Collocation. CC BY 4.0. -->

**この6手順は "書き始める前" と "書き直す前" の両方で必ず実行する。**  
linter は語彙の選択ミスを検出できない。判定できるのは書き手だけ。

### 手順 1 — 偏愛語を疑う

以下の語が文に登場したら **いったん手を止める**:

| カテゴリ | 典型的な偏愛語 |
|---|---|
| 動詞系 | 落ちる・壊れる・刺さる・効く・回る・動く・走る |
| 名詞系 | 設計・仕組み・構造・本質・流れ・全体像 |
| 形容詞・副詞系 | 解像度が高い・手触り・深い・シンプルな・綺麗な |

「設計」「構造」はコーディング教材で正当に使える語だが、抽象的な概念（情報・理解）に被せると擦り倒しになる。

### 手順 2 — 対象に実体があるか判定する

| 実体あり（OK） | 実体なし（擦り倒しの疑い） |
|---|---|
| 「DBスキーマ設計」 → 実際のテーブル定義がある | 「情報設計」→ 何の情報？どのテーブル？ |
| 「サーバーが落ちる」→ プロセス終了が実体 | 「体験が壊れる」→ 何がどう変わる？ |
| 「コンポーネントが回る」→ 描画サイクルが実体 | 「本質がわかる」→ どの動作が変わる？ |

**判定ルール**: 名詞の指示対象を1文で説明できるか？　できなければ擦り倒し。

### 手順 3 — NGなら比喩に逃げず具体へ書き直す

```
❌ 「この情報が落ちると設計が壊れる」
✅ 「userId が undefined になると、Prisma クエリの WHERE 句が正しく組み立てられずエラーが発生する」
```

**禁止パターン**: 比喩でごまかす（「まるで積み木が崩れるように…」）。具体の動作に翻訳する。

### 手順 4 — 構文の癖を直す

| 癖のある構文 | 直す方向 |
|---|---|
| 「〜は、〜だった」（過去形で語る） | 現在形 or 命令形へ（「〜になる」「〜してください」） |
| 「AではなくB」の対比が1文内に同居 | 2文に分ける |
| 「〜という」の入れ子 | 「〜とは」で定義文に |
| 「〜のようなもの」 | 実体で定義する |

### 手順 5 — 動詞・助詞・形容詞の選択ミスを正す

慣用句の破壊に注意（「課題を改善する」→「課題を解決する」「改善策を実施する」）。

| NG | OK |
|---|---|
| 「問題を改善する」 | 「問題を解決する」 |
| 「知識を身につける」（教材で頻出） | 「〜できるようになる」（行動目標に変換） |
| 「うまく動く」 | 「エラーなく起動し、〇〇を返す」 |
| 「ちゃんと設定する」 | 「.env に DATABASE_URL を記述する」 |

### 手順 6 — 重言・翻訳調を消す

| 重言 | 翻訳調（英語直訳臭） |
|---|---|
| 「まず最初に」→「まず」 | 「〜することができます」→「〜できます」 |
| 「約〜程度」→「約〜」 | 「〜という点に注意」→「〜に注意」 |
| 「返り値を返す」→「返す」 | 「行うことが必要です」→「必要です」 |

## レジスタ注意（教材と外部発信の違い）

- **教材の地の文** は敬体（です・ます）で軽いカジュアルさを持つ（初心者が親しみやすいよう）。親しみ表現は許容するが、AI臭・翻訳調・偏愛語は排除する。
- **global `ja-anti-ai-tone`** は PR/Issue/Slack 等の**外部発信文（敬体寄り）専用**。教材 md には適用しない。両者はレジスタが異なる。

---

## 前提

- 教材は「完全な初心者が挫折せずに完走できること」が唯一の成功基準
- AIが書いた不自然な日本語（文体ブレ・誇張・絵文字・省略）は販売品質に直結する欠陥
- ゲートは「通ってから出す」ではなく「通るまで書き直す」

## 執筆後の必須3ゲート（順番厳守）

### Gate 1: 文章衛生チェック（textlint）

```bash
npx textlint material/30days-curriculum/dayXX_xxx.md
```

**主な検出項目（外部ルール、自動修正可）:**
- `no-mix-dearu-desumasu`: だ・である調とです・ます調の混在
- `sentence-length`: 1文120文字超過
- AI臭フレーズ: 革命的・ゲームチェンジャー・究極・etc.
- 絵文字リスト (`**太字:**` 箇条書き)

**修正後再実行→エラー0になるまで続ける。**  
`--fix` オプションで一部自動修正可能:

```bash
npx textlint --fix material/30days-curriculum/dayXX_xxx.md
```

例外が必要な場合（コード例内の文体など）:
```markdown
<!-- textlint-disable no-mix-dearu-desumasu -->
コード例内のコメントはだ・である調で書く
<!-- textlint-enable no-mix-dearu-desumasu -->
```

---

### Gate 2: 品質チェック（check_quality.sh）

```bash
bash script/check_quality.sh material/30days-curriculum/dayXX_xxx.md
```

**チェック項目:**
1. 視覚化（表・図・スクショ）
2. コードブロック長さ（25行以下）
3. ステップ連続性（省略なし）
4. 禁止ワード（「同様に」「残りも」「以下略」etc.）
5. 技術スタック（shadcn/ui必須・MUI禁止）
6. コード完全性（filepath:コメント・省略禁止）
7. 理解度（注釈なし専門用語5個超 → FAIL）

**全Step PASS になるまで修正して再実行。**

---

### Gate 3: 初心者シミュレーション（冷えた別エージェント）

1日分完成後、**自分が書いた内容を忘れた別のエージェント**に通す。

**実行プロンプト（別エージェントに渡す）:**

```
あなたはプログラミング完全初心者です。
以下の条件を前提にこの教材を読んでください:
- HTMLとCSSは少し触ったことがある程度
- JavaScriptは変数・if文・for文が辛うじてわかる
- Node.js・npm・GitHubを一度も使ったことがない
- エラーメッセージを読む習慣がない

以下を **行番号付きで** 報告してください（出なければ「なし」と明記）:
1. 詰まりポイント（最低3個。「なぜ詰まるか」を具体的に）
2. 注釈なしで登場した専門用語（行番号と用語名）
3. 「次のStepに進んでいいか」確認できない箇所
4. 「ここで何をやっているか」がわからない箇所
5. エラーが出た時の対処法が書かれていない箇所
6. 偏愛語の擦り倒しが疑われる箇所（行番号付きで列挙。「〇〇行：『設計が壊れる』→実体が不明」の形式）

詰まりポイントが3個以上出た場合のみ「レビュー成立」とみなす。
0個は「読み込みが浅い」とみなしてやり直し。

---対象教材---
[教材内容をここにペーストする]
```

**判定基準:**
- 詰まりポイント 3個以上 → レビュー成立 → 指摘を全て修正してGate 1に戻る
- 詰まりポイント 0個 → レビュー不成立 → プロンプトを見直して再実行
- **偏愛語の擦り倒しが1件でも列挙された場合** → 書く前の6手順（手順1〜3）で書き直す → Gate 1に戻る

## 5つの不在（思考の問題として理解する）

<!-- Source: hana652/tech-writing-pack guide 03 — Anti-AI 5 Absences. CC BY 4.0. -->

**重要**: 5つの不在は「禁止表現のリスト」ではなく**思考の欠落**を指す。モグラ叩き（表現を潰す）ではなく、根本（なぜその思考が生まれるか）を理解する。

1. **態度の不在** — 書き手が自分の主張を持っていない。「〜かもしれません」「〜と思います」は思考放棄のシグナル。修正：「〜します」「〜になります」と断言する。

2. **勇気の不在** — 難しいことを難しいと言わない。「簡単です」「ご存知の通り」で難易度を隠す。修正：「ここが一番難しい部分です。〇〇に注意してください」と正直に書く。

3. **省略の不在** — 「同様に」「残りも」「〜省略〜」で思考を端折る。初心者には端折りが最も致命的。修正：全手順を書く。

4. **リズムの不在** — 1文が長すぎる（120字超）か短すぎて文脈が切れる。思考の塊と文の区切りが一致していない。修正：1文1意（一文で1つのことだけ伝える）。

5. **沈黙の不在** — 初心者が必ず疑問に思う「なぜ？」に答えていない。「こう書きます」の後に「なぜなら〜」がない。修正：理由を必ず添える。

詳細と症状パターン①〜⑮ → `hana652-tech-writing-pack/guides/03-anti-ai.md` 参照。

## コミット前確認

```bash
# 全ゲートを一括確認
npx textlint material/30days-curriculum/dayXX_xxx.md && \
bash script/check_quality.sh material/30days-curriculum/dayXX_xxx.md && \
echo "全ゲートPASS — コミット可"
```

lint-staged が git commit 時に自動でGate 1 + Gate 2を実行する。  
Gate 3（初心者シミュレーション）は人間/エージェントによる確認が必要。

# 06. AIに書き直させるプロンプト集

AIに書かせた下書きを「人間っぽく」整えさせるためのプロンプト集。
`03-anti-ai.md` の症状①〜⑭と `04-collocation.md` のコロケーション崩れを回避させる。コピペで使えます。

---

## 小技プロンプト（短い・差し込み式）

会話の途中で「ちょっと違うな」と感じたら投げる1行プロンプト。

### 1. スタンスを取らせる
```
中立的な意見はいらないので、スタンスを取って言い切ってください。
「ケースバイケース」「状況によります」は禁止。条件を切ってそれぞれ言い切ってください。
```
「どっちつかず感」を一気に消す。AIは断言を避けて中立に逃げがちなので、明示的に指示する。

### 2. 文末リズムを散らす
```
文末が単調にならないようリズムを持たせてください。「〜です。」が連続しないよう、
断定・推量・否定・問いかけ・体言止めを混ぜてください。
```
症状⑥（同一パターンの反復）対策。

### 3. 章境界アナウンスを禁止する
```
セクション冒頭で「ここから〜」「ここまで〜してきました」「次は〜していきます」のような
章境界アナウンスを使わないでください。章の境界は見出しが知らせています。本題から始めてください。
```
症状⑫対策。

### 4. テンプレ比喩を使わせない
```
比喩で、地図・仕様書・羅針盤・土台・柱・栄養・筋トレ・DNA・車の両輪・潤滑油・エンジン・
スパイス・レシピは使わないでください。具体例で直接書くか、別の比喩にしてください。
```
症状⑬対策。

### 5. 抽象動詞での提案を禁止する
```
提案やToDoで「効率化する/改善する/最適化する/強化する」のような抽象動詞は使わないでください。
「何を、いつまでに、どのレベルまで」が書けない動詞は逃げです。具体動作で書いてください。
```
症状⑨対策。

### 6. コロケーション崩れを直す（このパック独自）
```
以下の文章で、語と語の結びつき（コロケーション）が不自然な箇所を直してください。
意味と事実は変えないこと。特に次をチェック：
- 動詞と目的語のミスマッチ（例「課題を改善する」→「課題を解決する」）
- 自動詞・他動詞の取り違え（例「速度を向上する」→「速度を向上させる」）
- 形容詞と名詞のミスマッチ（例「濃い理解」→「深い理解」）
- 重言（例「まず最初に」「throwを投げる」）
- 翻訳調（例「影響を作る」→「影響を与える」）
修正箇所を変更前→変更後で一覧にし、その後に直した全文を出してください。
```
`04-collocation.md` 対策。AI日本語の不自然さに最も効く。

## まとめプロンプト（書き換え用・本格運用）

AIに書かせた下書きを丸ごと渡して「人間っぽく」書き直させるプロンプト。症状①〜⑭とコロケーション崩れを網羅的に潰す。

```
# 役割
あなたは日本語の編集者です。以下の「元の文章」はAIが書いた下書きです。
意味と事実関係は変えずに、人が書いたと読まれる自然な日本語に書き直してください。追加の質問はしないでください。

# 厳守ルール（内容）
- 内容の捏造はしない。数字・固有名詞・コードを足さない
- 前置き宣言（「結論から言うと」「本記事では」「以下で解説します」）は入れない。本題から書き出す
- 安全クッション（「一般的に」「多くの場合」「状況によって異なります」「一概には言えませんが」）は削除する
- 抽象語だけで押し切らない。元文の範囲で動詞中心に具体化する
- 同義語の言い換え連打（重要・大切・欠かせない）はやめ、1回で言い切る
- 「まとめると」「要するに」「総じて」の抽象まとめの繰り返しは削る
- 文長を均一にしない。短文と長文を混ぜる。同じ型の連続を避ける
- 接続詞は必要最小限。「さらに」「また」「したがって」「そのため」は順接の場合ほぼ削れる
- 章境界アナウンス（「ここから〜」「ここまで〜してきました」「次は〜していきます」）は使わない
- 「ステップ1：」「STEP 1：」のラベル列挙はしない。番号付きリストにする
- テンプレ比喩（地図・仕様書・羅針盤・土台・柱・栄養・筋トレ・DNA・車の両輪・潤滑油・エンジン・スパイス・レシピ）は使わない
- 提案で抽象動詞（効率化/改善/最適化/強化する）で逃げない。「何をいつまでにどのレベルまで」を書く

# 厳守ルール（コロケーション）
- 動詞と目的語のミスマッチを直す（課題を改善する→解決する、目標を成功する→達成する、効果を与える→発揮する）
- 自動詞・他動詞を正す（速度を向上する→向上させる、エラーを発生する→発生させる）
- 形容詞と名詞を合わせる（濃い理解→深い理解、高い課題→大きな課題）
- 重言を消す（まず最初に→まず、throwを投げる→例外を投げる、約〜程度→約〜）
- 翻訳調を和らげる（影響を作る→影響を与える、決定を作る→判断する）

# 厳守ルール（記号・表記）
- Markdown装飾（**太字**、##見出し、箇条書き記号）の残骸を消す（※コードブロック内は除く）
- 「」を多用しない。「」の中に『』を入れ子にしない
- ()を多用しない。補足は本文に組み込む。()の中で責任回避（「ただし状況による」）をしない
- コロン「：」+半角スペースは使わない。／（全角スラッシュ）で並列しない。「、」で並べる
- —（em-dash）、──（罫線ダッシュ）は使わない
- 「参考になれば幸いです」「まずは小さく始めましょう」などの締めの定型句を入れない

# 出力形式
- 書き換え後の文章だけを出力する（解説・前置きは出さない）
- 元の段落構成は大きく崩さない。文章量は元文から±20%以内
- コードブロックの中身は一切変更しない

---

## 元の文章
<<< ここに元の文章を貼る >>>
```

## CARTEフレームワーク（執筆・調査依頼の入力テンプレ）

AIに依頼するとき、毎回チェックすべき5項目。神プロンプト集はモデルが変わると陳腐化するが、CARTEは普遍的に効く「指示出し力」の型。

| 項目 | 内容 | 効果 |
|---|---|---|
| **C** Context（背景と目的） | なぜそれをやるのか、何に使うのか | 一般論への逃げを止める |
| **A** Audience（相手） | 誰が読むのか、どれくらい詳しい人か | 読者ズレを防ぐ |
| **R** Role（役割） | AIに何の役割を演じさせるか | 補助輪 |
| **T** Tone（トーン） | 文体・温度・距離感 | 補助輪 |
| **E** Example（例・完成形） | 構成、見出し、形式の見本 | 出力安定化。CARTEで最強の効きどころ |

### CARTEで書く例（技術記事の依頼）

```
[C] 背景: Next.jsのApp Routerに移行したらメタデータの設定でハマった。
    同じ所で詰まる人向けに記事を書きたい
   目的: 検索で来た人がコピペで解決できる状態にする

[A] 相手: Next.jsの基本はわかるが、App Routerは初めての人
   詳しさ: Reactは書ける。SSR/メタデータの細かい挙動は知らない
   重視: コピペで動くコードと、なぜそうなるかの最小限の説明

[R] 役割: 実務経験のある技術ライター

[T] トーン: ですます調、淡々と。煽らない。前置きは短く

[E] 完成形: トラブル解決型。リードで原因を即提示 → 変更前→変更後のコード →
    原因の説明 → ケース別解決策 → 確認チェックリスト。コードは全て動く形で
```

特に **C・A・E** が効く。R・T はブレを減らす補助輪。

## 逆質問プロンプト（曖昧な依頼を埋める）

依頼が曖昧なまま手を動かさせると後で破綻する。**成果物を作らせる前に、AI側に確認質問させる**フェーズを挟むと手戻りが激減する。

```
1. いきなり成果物を書かないでください。最初は確認質問だけしてください（最大5問）。
2. 質問は「依頼に関係があるか」「具体に絞れているか」「作業を前に進めるか」の3軸で品質を担保してください。
3. どうしても推測で埋めるなら、その部分に「仮置き」と明示してください。
```

なぜ効くか：AIはデフォルトで「曖昧でも勝手に推測して答える」癖がある。確認フェーズを分離すると精度が上がる。質問数に上限（5問）を置かないと暴走する。

## 「直し方」プロンプト（出力のズレを言語化する）

プロンプトの上達を分けるのは「最初の指示」より「直し方」。「なんか違う」で止まると伸びない。**ズレを言語で切る**と短い反復で精度が上がる。

### NG（地獄のループ）
「もう少し良くして」「なんか違う」「もっとちゃんと」

### OK（軸で切る）
- 「**抽象度が高い**ので、もっと固有名詞とコード例を入れて」
- 「**前提が抜けている**。〇〇という環境を補ってから書き直して」
- 「**対象読者のレベルが合っていない**。初学者でもわかる言葉に落として」
- 「**論点が散っている**ので、〇〇に絞って書き直して」

### 評価軸チェックリスト（ズレを言語化するフレーム）
直しを依頼する前に、どれに当たるか自分で言語化する：
抽象度（高/低）／前提（足りない/余計）／対象読者（レベル違い）／論点（散っている）／トーン（堅/カジュアル）／構造（順序・重複）／量（長/短）／具体性（数字・コード不足）

## 使い方の流れ

1. **執筆段階**：CARTEで依頼してAIに下書きさせる（曖昧なら逆質問プロンプトを先に）
2. **書き換え段階**：軽微なら小技プロンプト、全面的なら「まとめプロンプト」を別AIに投げて一括書き換え
3. **機械チェック**：`scripts/ai-smell-lint.py` で症状・コロケーションの候補を洗い出す
4. **最終確認**：人間が「自分の判断・実体験・動作確認したコード」を足して、`03-anti-ai.md` の5つの不在を埋める

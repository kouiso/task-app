# 01. 技術ブログのSEO

SEO（検索エンジン最適化）＝「検索した人に記事を見つけてもらうための設計」。
テクニックの前に、いちばん大事な原則を1つ。

> **SEOの9割は「誰の、どんな検索に、答える記事か」を決めること。**
> 小手先のキーワード詰め込みではなく、検索した人の疑問に正面から答える記事がいちばん強い。

技術ブログは特にこれが効く。読者は「エラーを解決したい」「やり方を知りたい」という明確な目的を持って検索してくるから、目的に正確に答えればちゃんと読まれる。

---

## 1. 検索意図（インテント）を見極める

検索キーワードの裏には、必ず「何がしたいのか」がある。これを**検索意図**と呼ぶ。技術系の検索意図は、だいたい4タイプ。

| タイプ | 検索の例 | 読者が欲しいもの | 合う記事型 |
|---|---|---|---|
| やり方を知りたい（How-to） | `react useeffect 使い方` | 手順とコード例 | チュートリアル/解説 |
| エラーを直したい（Fix） | `typescript cannot find module エラー` | 原因と具体的な解決策 | トラブル解決 |
| 違いを知りたい（Compare） | `npm yarn pnpm 違い` | 比較と選び方 | 比較記事 |
| 概念を理解したい（Learn） | `クロージャ とは` | わかりやすい説明 | 解説記事 |

**最初にやること**：書こうとしているキーワードがどのタイプか判定する。
タイプを間違えると（例：「エラーを直したい」人に概念解説を返す）、読まれず離脱される。

検索意図は実際にそのキーワードでGoogle検索して、**上位10記事が何を書いているか**を見れば一発でわかる。上位がコード中心なら読者はコードを求めている。

---

## 2. キーワードの選び方

技術ブログで個人や小規模サイトが勝つコツは、**「ビッグキーワードを避けて、具体的なロングテールを狙う」**こと。

```
❌ 狙わない（強すぎる）
「JavaScript」「React」「Python 入門」
→ 大手メディア・公式ドキュメントが独占。個人ブログは勝てない

✅ 狙う（具体的で競合が弱い）
「react useEffect 無限ループ 原因」
「python venv windows 有効化されない」
「next.js app router メタデータ 動的生成」
→ 困っている人が今まさに検索している。記事数も少ない
```

### ロングテールの見つけ方（無料でできる）

1. **サジェスト**：Googleの検索窓にキーワードを打つと出る候補。実際に検索されている言葉
2. **「他の人はこちらも質問」**：検索結果の中段に出るQ&A。そのまま見出しに使える
3. **関連検索**：検索結果の最下部
4. **自分のエラー体験**：自分がハマって検索したキーワードは、他の人も検索している
5. **GitHub Issues / Stack Overflow**：実際の質問文がキーワードの宝庫

> 個人技術ブログの王道は **「自分がハマったエラーや実装を、検索キーワードの形で記事にする」**。
> 実体験があるぶん内容が具体的になり、AI記事との差がいちばん出る領域。

---

## 3. 記事の構成（SEOで効く型）

検索流入を取る記事には、ほぼ共通の骨格がある。

```
タイトル        ← キーワードを含む。具体的に
リード文（導入） ← この記事で何が解決するか。3〜4行で結論の入口
本文            ← 見出し(H2/H3)で区切る。1見出し1トピック
  結論先行      ← 「答え」を先に。理由・手順は後
  コード例      ← コピペで動くものを。実行結果も示す
まとめ          ← 要点の再確認（任意。短く）
```

### タイトルの作り方

- **キーワードを左寄せで含める**（重要語を前に）
- **具体的にする**：「Reactの解説」より「React useEffectで無限ループする原因と直し方」
- **数字・バージョンを入れる**：「【Next.js 15】」「3つの方法」は具体性が出てクリックされやすい
- **「徹底解説」「完全ガイド」は避ける**：テンプレ語で中身が薄く見える（→ `03-anti-ai.md`）

### リード文（導入）が最重要

検索から来た人は、最初の3行で「この記事に答えがあるか」を判断して、無ければ即座に戻る。
だから**リードで結論の入口を見せる**。

```
❌ 弱いリード
「この記事ではuseEffectについて解説します。useEffectはReactのフックの1つで…」
→ 何が解決するか不明。前置きが長い

✅ 強いリード
「useEffectが無限ループする原因はほぼ1つ。依存配列にオブジェクトや関数を
直接入れていることです。この記事では原因の特定と、3パターンの直し方を
コード付きで示します。」
→ 結論の入口が見える。読者は「これだ」と判断できる
```

## 4. 技術ブログ特有のSEOポイント

### コードブロックは正しくマークアップする

```markdown
言語を指定する（シンタックスハイライトが効く）

​```javascript
const x = 1;
​```
```

言語指定（` ```js ` など）は読みやすさだけでなく、検索エンジンが「これはコードだ」と理解する助けになる。

### コピペで動くコードを載せる

import文の省略・前提条件の未記載は、技術記事で最も嫌われる。
**「そのままコピペして動くか」を自分で実行して確認する**。動かないコードは滞在時間を下げ、SEOにも悪影響。

### 公式ドキュメントへリンクする

公式（MDN、React Docs、各ライブラリのDocs）へのリンクは、記事の信頼性を上げる。
「公式はこう書いてあるが、実際はこうハマる」という構成は、公式に勝てる数少ないパターン。

### 更新日を入れて、古くなったら直す

技術記事は鮮度が命。バージョンが上がると情報が古くなる。
**公開日だけでなく更新日を表示し、定期的に手を入れる**。Googleは更新されている記事を評価する。古い記事を放置するより、1本を更新し続ける方が強い。

### 構造化データ（できれば）

`Article` や `FAQPage` の構造化データ（JSON-LD）を入れると、検索結果でリッチに表示されることがある。
最初は気にしなくていい。記事が増えてから取り組めば十分。

---

## 5. 内部リンク（記事同士をつなぐ）

記事が3本以上たまったら、**関連する記事同士をリンクでつなぐ**。

- 「useEffectの記事」から「useStateの記事」へ
- 入門記事から応用記事へ、応用記事から入門記事へ（双方向）

内部リンクの効果：
1. 読者が次の記事に回遊する（滞在時間が伸びる）
2. 検索エンジンがサイト内の関連性を理解する
3. 「このテーマに詳しいサイト」と評価されやすくなる（トピッククラスター）

> 技術ブログは**1テーマを深く掘る**と強い。
> 「React関連10記事」が相互リンクで束になっていると、バラバラの30記事より評価される。

---

## 6. 計測（書いたあと）

公開して終わりではない。最低限この2つは入れる。

| ツール | 何がわかるか | 費用 |
|---|---|---|
| **Google Search Console** | どのキーワードで表示・クリックされたか、検索順位 | 無料 |
| **Google Analytics 4 (GA4)** | アクセス数、滞在時間、どこから来たか | 無料 |

### Search Consoleの見方（最初はこれだけ）

1. 「検索パフォーマンス」を開く
2. **表示回数は多いがクリックが少ないキーワード**を探す → タイトルを改善する余地
3. **順位が11〜20位のキーワード**を探す → あと一押しで1ページ目。その記事を加筆する

順位を上げる近道は、新記事を量産することより、**「あと少しで上位の記事」を加筆強化する**こと。

---

## 7. やってはいけないこと（ペナルティ・逆効果）

| NG | なぜダメか |
|---|---|
| キーワードの詰め込み | 不自然な文章は順位が下がる。今は逆効果 |
| 中身の薄い記事の量産 | 質の低いページはサイト全体の評価を下げる |
| 他サイトのコピペ・AI丸投げ | 独自性ゼロ。コードの動作未確認は信頼を失う |
| 動かないコードを載せる | 滞在時間が落ち、二度と読まれない |
| タイトルと中身が違う（釣り） | 即離脱され、SEOにマイナス |

---

## まとめ：技術ブログSEOの優先順位

1. **検索意図に正確に答える**（これが9割）
2. **自分がハマった具体的なキーワードを狙う**（ロングテール）
3. **リードで結論の入口を見せる**
4. **コピペで動くコードを載せる**
5. **1テーマを深掘りして内部リンクで束ねる**
6. **Search Consoleで「あと一歩」の記事を加筆する**

小手先のテクニックより、**「自分が困って検索したことを、次に困る人のために正確に書く」**。これが技術ブログでいちばん強いSEOです。

# 03. AI臭除去ガイド

テクニカルライティング（`05`）とは別の軸。
- テクニカルライティング ＝「正確に、無駄なく伝わるか」
- このガイド ＝「読んでいて違和感がないか。人間が書いたように読めるか」

両方を満たして初めて、まともな技術記事になる。
AIに下書きさせるとスピードは上がるが、そのままだと「AIっぽさ」が残る。その正体と消し方を扱う。

---

## なぜAI文章は「AI臭い」のか

NG表現をリスト化して潰しても、別の表現で同じ臭さが出る。モグラ叩きになる。
根本原因は、AIの文章に **「5つの不在」** があるから。表現の問題ではなく、思考の問題。

---

## 5つの不在

### 1. 態度の不在

AIは対象に態度を持たない。好きでも嫌いでもない。だから全部のトピックを同じ温度で書く。

人間は違う。大事だと思うところは力が入るし、どうでもいいところは雑に流す。「このライブラリは正直おすすめしない」と思って書く人と、「こういうライブラリがあります」と並べるだけの人では、同じ情報でも文章の体温が違う。

**対処**：書く前に「このテーマのどこが面白い / ハマる / 重要なのか」を1行で言語化する。その1行が記事全体の態度になる。

```
❌ 態度がない
「ReactのuseEffectについて解説します。」
→ この1文から書き手の態度が見えない

✅ 態度がある
「useEffectの無限ループ、9割は依存配列のミスです。やりがちなやつから潰します。」
→ 「みんなここでハマる」という態度が出ている
```

### 2. 勇気の不在

AIは言い切らない。「〜と言えるでしょう」「〜の可能性があります」「〜かもしれません」で保険をかける。

人間のライターも間違いたくはないが、読者のために言い切る。「これは使わない方がいい」「ここが一番大事」と判断を示す。判断を示さない記事は、読者にとって価値がない。

**対処**：各セクションで「結局どうなのか」を言い切る文を最低1つ入れる。断言できない場合は「ケースバイケース」と逃げず、「こういう条件ならこう、こういう条件ならこう」と条件を切ってそれぞれ言い切る。

```
❌ 保険をかけている
「状況によってはuseMemoが有効な場合もあると言えるでしょう。」

✅ 言い切っている
「useMemoは、計算が重くて再レンダリングが頻繁なときだけ使う。それ以外は不要です。」
```

### 3. 省略の不在

AIは全部書こうとする。聞かれていないことまで網羅する。「さまざまな」「包括的な」「網羅的に」はその表れ。

人間のライターは「これは今回いらない」と判断して省く。省くには何が重要かの判断が要る。

**対処**：各セクションの下書きを書いた後、「ここを読み飛ばされても記事の主旨は伝わるか」をテストする。伝わるなら削るか短くする。削ることへの恐怖を捨てる。記事の価値は情報量ではなく、読者が持ち帰る判断材料の質で決まる。

### 4. リズムの不在

AIの文章は、文の長さが均一になりやすい。30〜50字の文がずっと続く。段落もセクションも均一。

人間の文章にはリズムがある。長い文の後に短い文。3段落の説明の後に1文だけの段落。この緩急が「読みやすさ」をつくる。

**対処**：
- 長い文（40〜50字）の直後に短い文（10〜20字）を置く
- 3段落以上続いたら、1文だけの短い段落を挟む
- セクションの長さを意図的に不均一にする。大事なセクションは長く、補足は短く

```
❌ リズムがない（均一な文長）
「非同期処理はPromiseを使って書くことができます。
Promiseはresolveとrejectの2つの状態を持っています。
async/awaitを使うとより簡潔に書くことができます。」

✅ リズムがある（長短の緩急）
「非同期処理は、まずPromiseを理解するところから。

Promiseには成功と失敗の2状態しかない。これだけ。

async/awaitはPromiseの書き方を変えただけで、
中身は同じPromiseが動いています。」
```

### 5. 沈黙の不在

AIは黙れない。すべての段落に結論を書き、すべてのセクションにまとめを入れる。余韻がない。読者が自分で考える余地を残さない。

人間のライターは、あえて結論を書かないことがある。問いだけ投げる。「あなたならどう設計します？」で段落を閉じる。この「間」が読者の思考を促す。

**対処**：記事全体で1〜2箇所、「答えを書かずに問いで終わるパラグラフ」を入れる。設計の是非など答えが割れるテーマでは、結論を出すより問いを残すほうが誠実なこともある。

## なぜAIはこう書くのか（7つの原因と対抗ルール）

症状の上流にある「なぜAIはそう書いてしまうか」。LLMの仕組みを知ると、症状を根本から避けられる。

### 原因①：平均への収束（mode collapse）
- **仕組み**：LLMは「次に来る確率が最も高い語」を選ぶ。だから最もありがちな表現・テンプレに収束する
- **生む症状**：画一的・無難・テンプレ（症状⑬テンプレ比喩、①ヘッジ）
- **対抗ルール**：一番ありがちな言い方を捨て、2〜3番目の表現を選ぶ。固有名詞・具体的な数字を入れて「平均」から外す

### 原因②：RLHFによる均質化（homogenization）★最重要
- **仕組み**：人間フィードバック学習で「親切・網羅的・丁寧・無害・中立」が高報酬になる。評価者は長い回答を高品質と取り違える（length bias）、ユーザー同調を好む（sycophancy）
- **生む症状**：ヘッジ・両論併記・冗長・箇条書き偏重（症状①⑪⑤③）
- **対抗ルール**：言い切る。両論併記でなく態度を1つ出す。網羅を捨てて1点に絞る。短くても言い切れていればよい

### 原因③：学習データの文体偏り（register）
- **仕組み**：Web上のSEO記事・ビジネス書・自己啓発・マーケコラムを大量学習し、その「読ませる型」を"良い文章"として内在化している
- **生む症状**：キャッチコピー構文・煽り・定型章立て（症状⑧否定対比、②メタ発言、⑬）
- **対抗ルール**：「AではなくB」構文を禁止。「いかがでしたか」「実は」「ぜひ」等のテンプレ定型を禁止

### 原因④：身体性・固有体験の欠如★
- **仕組み**：AIには書き手の記憶・体験・固有のエピソードがない。だから具体を出せず、抽象・一般論に逃げる
- **生む症状**：具体性の欠如、抽象語、一般論（症状⑩エモ抽象化、①）
- **対抗ルール**：固有のエピソード・実際のエラー文・固有名詞・数字を必ず入れる。「ハマるかもしれません」と説明せず、具体的な場面で描写する

### 原因⑤：逐次生成の自己拘束（autoregressive commitment）
- **仕組み**：LLMは左から1語ずつ生成し、後戻りできない。破綻を防ぐため、先に「3つあります」と枠を宣言してから埋める戦略を取りやすい
- **生む症状**：個数の予告・ナンバリング偏重・予告口調・作業実況（症状②）
- **対抗ルール**：「理由は3つあります」式の個数予告を禁止。いきなり中身に入る。「まず〜する」の作業実況見出しを書かない

### 原因⑥：format collapse（整然＝良いと学習）
- **仕組み**：指示追従の微調整で「読みやすさ＝構造の整然さ」を学ぶ。構造化テンプレ（箇条書き・ナンバリング）への依存が"多様性崩壊"を起こす
- **生む症状**：均等な段落・過剰なブロック化・箇条書き濫用（症状③、②）
- **対抗ルール**：箇条書きを文章に戻せないか考える。段落の長さを揃えない。視覚要素は1セクション1つまで

### 原因⑦：burstiness（文長の揺らぎ）の低下★
- **仕組み**：人間は短い文と長い複雑な文を交互に使い、文長の揺らぎ（burstiness）が大きい。AIは均一になりやすく、AI検出器もこの「揺らぎのなさ」を指標にする
- **生む症状**：文長が均一・リズムが平板（症状⑥反復、不在4「リズム」）
- **対抗ルール**：短文と長文を意図的に混ぜる。語尾を散らす。たまに体言止め・1文だけの段落を入れる

### 特に効く4つの対抗ルール
1. **文長にわざとデコボコを作る**（burstinessを上げる＝原因⑦）
2. **「3つあります」式の個数予告を禁止**（原因⑤）
3. **網羅をやめて1点に絞る・長さで誤魔化さない**（原因②）
4. **一番ありがちな言い方を捨て、具体で平均から外す**（原因①④）

## 症状パターン（①〜⑭）

5つの不在が表面化したもの。多くは機械的にチェックできる（`scripts/ai-smell-lint.py`）。

### 症状①：ヘッジ（不在2「勇気」）
断定を避けて逃げ道を作る。「もちろん」「とはいえ」「一方で」「〜と言えるでしょう」が1文に複数。

```
AI: 「もちろん学習コストはありますが、一方で生産性は上がると言えるでしょう。」
→  「学習コストはある。でも慣れれば生産性は上がります。」
```

### 症状②：メタ発言・作業実況（不在3「省略」）
内容ではなく記事の構造を説明する。「〜を解説します」「〜を見ていきましょう」。見出しや目次でやれば済む。

```
AI: 「ここからが本題です。順番に見ていきましょう。」
→  （いきなり本題に入る）
```

**作業を実況する見出しも同じ症状**。「まず環境構築する」「〜を整理する」のように手の動きを実況する見出しはAI臭が強い。見出しは「何をするか」でなく「何が言えるか（結論・内容）」で書く。

```
AI: H2「まず原因を調べる」 → 内容が無い作業実況
→  H2「原因は依存配列のオブジェクト比較」（中身を言い切る）
```

**「ステップ」「STEP」連発も同じ**。番号付きリストにすればいい内容を、本文で「ステップ1：」と宣言するのは説明書のテンプレ。

### 症状③：均等・対称への執着（不在1「態度」＋不在3「省略」）
全セクション同じ長さ、同じ構造。メリット3つ書いたらデメリットも3つ。人間が書くと大事なところは長く、そうでないところは短い。

### 症状④：過剰な共感演出（不在1「態度」）
「〜ですよね」「〜ってなりますよね」を連発するが、具体的な経験に基づかないから空っぽに見える。

```
AI: 「環境構築、面倒ですよね。でも大丈夫です。」
→  「環境構築でnodeのバージョン違いに3回ハマりました。volta入れたら解決した。」
```

### 症状⑤：網羅欲（不在3「省略」）
聞かれていないことまで全部カバーしようとする。「さまざまな」「包括的な」「網羅的に」は網羅できていないときのごまかし。

### 症状⑥：同一パターンの反復（不在4「リズム」）
語尾・文構造・セクション構成が無意識にループする。

- セクション冒頭の「〜について説明します」連打
- 「〜です。〜です。〜です。」の語尾3連続
- 全セクション同じ構造（定義→説明→注意点→まとめ）の繰り返し

### 症状⑦：助詞省略のキャッチコピー構文（不在1＋不在4）
「単語、〜してほしい」のように助詞（は・が・を）を省略して読点でつなぐ。スライドやキャッチコピー風になる。

```
AI: 「このコード、リファクタしてほしい。」
→  「このコードはリファクタした方がいい。」
```

1文だけなら修辞効果を狙えるが、AIは無自覚に連発する。

### 症状⑧：否定→肯定の対比構文（不在2＋不在5）
「〜じゃない。〜だ。」の形で格言風に決める構文。AIが「かっこいい一言」を出そうとするときに多用。

```
AI: 「大事なのは言語じゃない。設計だ。」
AI: 「動かないのは才能がないからじゃない。基礎が抜けているからだ。」
→  「動かない原因は、たいてい基礎の理解が抜けているところにあります。」
```

1記事に1回なら効果的だが、AIは何度も使う。中身が薄いときほど使いがち。

**派生：文中の「AではなくB」構文**も同じ臭い。AとBを1文に詰め込まず、分けてそれぞれ独立した主張として立てる。

**例外**：仕様書・エラーメッセージ・引用の原文は改変しない。

### 症状⑨：構造化動詞（不在3＋不在4）
「〜を仕組みにする」「〜を設計する」「〜を構造化する」「〜を可視化する」「〜を最適化する」のように、具体的な動作を抽象動詞でまとめる構文。

```
AI: 「エラー処理を仕組みにする。」
AI: 「開発フローを最適化する3つのコツ。」
→  「try-catchで例外を1箇所にまとめ、ログに出す。」（具体）
→  「PRを小さく分ける、CIを通す、レビューを24時間以内に返す。」（具体）
```

抽象動詞は「具体を書く労力の回避」。本当にやることが決まっているなら、具体的な動作・対象で書ける。

**特に注意**：仕組みにする/設計する/構造化する/可視化する/最適化する/抽象化する/言語化する/アップデートする

**ビジネス文書頻出の抽象動詞**（提案・ToDoで逃げに使われる）：
- 「効率化する」→「ビルド時間を2分から30秒に短縮する」
- 「改善する」→「APIの応答を500msから100ms以下にする」

「何を、いつまでに、どのレベルまで」が書けないなら、その動詞は逃げている。

### 症状⑩：エモ抽象化（不在1＋不在3）
「〜の温度」「〜の解像度」「〜の手触り」「〜の質感」のように、感覚っぽい抽象名詞で雰囲気を演出する構文。

```
AI: 「コードの解像度を上げる。」
AI: 「設計の手触りが良い。」
→  「変数名を具体的にして、関数を10行以内に保つ。」（具体）
```

語彙自体が悪いのではなく、具体を書かずに済ませる便利語として乱用されるのが問題。1記事に1個まで。

### 症状⑪：ぼかし語尾と状況プレゼン（不在2）
断言を避けるため、状況を観察する体で逃げる。

```
AI: 「今、開発の現場で大きな変化が起きている。」
AI: 「これは、ある意味で設計の問いなのかもしれない。」
→  「2025年以降、型安全を重視するチームが増えた。」（具体的な観察）
→  「ここから先は、あなたのチームの方針次第です。」（断言）
```

**特に注意**：「〜が起きている」「〜が問われている」「〜という話」「〜なのではないか」「〜という名の〜」

### 症状⑫：章境界での予告・回顧アナウンス（不在3＋不在4）
セクションの冒頭・末尾で「ここから〜」「ここまで〜してきました」「次は〜していきます」と章の境界をいちいち音声案内する。

章の境界は**見出し（H2）がすでに知らせている**。本文で重ねて予告するのは冗長。

```
AI: 「ここまでがセットアップです。ここからは実装を見ていきます。」
AI: 「整理します。」「まとめます。」
→ 削除する。または「で、実装はどう書くか。」と短い問いで繋ぐ
```

**対処**：章境界はH2に任せる。繋ぎが要るなら「で、〜」「ところで、〜」で済ます。章末の余韻は「問い」か「短い断言」で。

### 症状⑬：テンプレ比喩の濫用（不在3）
AIがビジネス記事・自己啓発記事を書くと、特定の比喩を高頻度で使う。

特に頻出：地図／仕様書／設計書／羅針盤（コンパス）／土台／柱／栄養／筋トレ／DNA／車の両輪／潤滑油／エンジン／スパイス／レシピ

```
AI: 「テストはコードの設計書である。」
AI: 「ドキュメントは開発の潤滑油だ。」
→  「テストがあると、リファクタしても壊れていないか1秒で確認できる。」（具体）
```

使いたくなったら「この概念は本当に地図と同じ構造か？」を自問する。多くの場合、直接書いた方が伝わる。

### 症状⑭：過去形の乱用（不在2）
現在も成立している事実を過去形で書く癖。AIが「分析の結果として発見した」体で書くときに多用。

```
AI: 「調べてみると、原因は依存配列にあった。」
AI: 「比較すると、A群はB群より速い傾向が見られた。」
→  「原因は依存配列にあります。」（現在形で断言）
→  「AはBより速い。」
```

現在も真である内容は現在形で書く。「〜だった」「〜がわかった」「〜が見られた」が連続していないか確認。

### 症状⑮：過度な短文化・主語/目的語の省略・常体落ち（不在4「リズム」の逆作用）

「AIっぽさ＝文が長い」と誤解して、一文を短く切りすぎると、別のAI臭が出る。むしろ**短文の連打・ブツ切りはAI検出されやすい**。AI臭を消そうとした結果、逆にAI臭くなる典型パターン。

- **主語・目的語の過度な省略**：「具体に戻す。」「言い切る。」のように主語も目的語も無い体言止め・命令形が続く。短くしようとして、必要な要素まで削っている
- **常体落ち（文体の不統一）**：ですます調の中に「〜だ」「〜する」「体言止め」がランダムに混じる（逆もある）。人間は文体を最後まで統一する

```
❌ 短くしすぎ・主語省略・文体ちぐはぐ
「直し方はシンプル。具体に戻す。言い換えない。人間に近づく。」

✅ 主語・目的語を残し、ですますで統一、自然な長さ
「直し方そのものはシンプルです。気取った言葉を見つけたら、別の言葉に
言い換えるのではなく、何を言いたいのかに戻してあげましょう。」
```

**対処**：
- 一文を無理に短くしない。短文と長文の「緩急」は付けるが、**緩急のために主語・目的語まで削らない**
- ですます調かである調か、どちらかに決めて最後まで貫く（→ `05-technical-writing.md` 「文体の統一」）
- 「短文化」と「主語省略」は別物。リズムは文の長さで作る。文の要素は削らない
- ※ この症状は機械検出が難しい（→ lintは拾えない）。目視で確認する

## 記号NG（AI生成時の痕跡）

AIの出力をそのままコピペすると、装飾記号が残る。これがいちばん「AIっぽく」見える要因の1つ。

| 記号 | なぜAI出力で残るか | 対処 |
|---|---|---|
| `**太字**` のアスタリスク | Markdownのレンダリング前が残る | 装飾は使わず本文に溶かす（※コードブロック内は対象外） |
| `—`（em-dash） | 英語圏の言い換え記号。日本語では「、」「。」で済む | 全削除。代わりに句点 |
| `——`／`──`（2文字ダッシュ・罫線） | 装飾系記号 | 全削除 |
| `：`（コロン）+ 半角スペース | 英語の `: ` 癖 | 「：」を消し、後の語を独立した文に |
| `／`（全角スラッシュ）で概念並列 | AIは「A／B／C」と書きがち | 「、」または「と」で並列 |
| `()` での補足過多 | （補足）（注釈）（ただし）を多用 | 本文に組み込むか削除 |

※ コードブロック・インラインコード（`` `code` ``）の中の記号は対象外。コードは原文のままが正しい。

### 語尾の反復チェック
同じ語尾が3回以上連続していないか。特に「〜ましょう」「〜でしょう」「〜です」「〜ません」。
対処は個数制限ではなく、語尾のバリエーション。体言止め・疑問形・倒置で散らす。

# 05. テクニカルライティング

「正確に、無駄なく伝わるか」だけを扱う。文体やAI臭は `03-anti-ai.md`、語の結びつきは `04-collocation.md` を参照。
技術記事は「正確さ」が命。コードと同じで、曖昧な説明は読者を確実にハマらせる。

---

## 1. 文の書き方

### 一文一意
1つの文では1つのことだけを言う。主語と述語の関係を2つ以上作らない。

### 文の長さ
50字が目安。60字超で分割を検討。80字が上限。
切り方：接続助詞（「が」「ので」「ため」）の手前で切る。

### 主語と述語
- 主語と述語を近づける
- 主語を文の途中で変えない（変わるなら文を分ける）
- 主述のねじれを排除する（「原因は〜ことが原因です」のような重複）
- 主語が前の文と変わるときは明示する

### 修飾語の配置（本多勝一の4原則）
1. 長い修飾語を先に、短い修飾語を後に
2. 節（主語＋述語を含む）を先に、句（短い語句）を後に
3. 大状況・重要内容を先に、小状況・補足を後に
4. 係る語と受ける語は近くに置く

### 接続助詞「が」と中止法
- 「が」は逆接以外（前置き・並列）で多用しない。文を切って接続詞でつなぐ
- 「〜し、」「〜であり、」で文をつなぎ続けない。句点で切る

### 同じ助詞の連続禁止
同じ助詞が1文に3回以上 → 書き換えを検討。
`❌ このコードの処理の流れの説明は` → `✅ このコードがどう動くかを説明すると`

### 受身形の制限
能動態で書けるなら能動態で書く。ただし行為主体を示す必要がない場合は受身形OK。
（例：「この関数は内部でキャッシュされます」は主体不要なので受身でよい）

### 二重否定の回避
`❌ 動かないわけではない` → `✅ 一応動く`

### 読点
意味が変わる位置には必ず打つ。基本位置：主語の後（長い文）、接続詞の後、条件節の後、並列語句の間、修飾のかかり先を明確にする位置。

## 2. 段落の書き方

- **一段落一トピック**：トピックが変わったら段落を改める
- **トピックセンテンス**：段落の冒頭に最も伝えたいことを置く。冒頭だけ拾い読みすれば全体の主旨がわかる状態が理想
- **段落の長さ**：3〜5文、150字程度。1文だけの段落は強調のみ。10文超は分割
- **結論先行**：結論 → 理由・根拠 → 具体例・補足
- **原則→例外の順序**：原則を先に述べ、例外を「ただし」で後に続ける

---

## 3. 削るべきもの

### 削っても意味が変わらない修飾語
基本的な / 具体的な / 一般的な / 〜的な / ある意味 / いわゆる
→ 使う前に「この語がなくても文意は同じか」を確認する。

### 冗長な定型パターン
| 冗長 | 簡潔 |
|---|---|
| 〜することができる | 〜できる |
| 〜ということです | 〜です |
| 〜を行う | 〜する（「実装を行う」→「実装する」） |
| 〜に関して / 〜において | 〜で / 〜の |
| 〜のほうが | 〜が |
| 〜という（不要な場合） | 削除 |

### 重複表現
`各〜ごとに` / `約〜くらい` / `まず最初に` → 片方を削る（→ `04-collocation.md` の重言）。

### 情報の多重掲載
1つの情報は原則1箇所。ただし冒頭概要と本文、本文とまとめの重複は構造上許容。
避けるべき：見出しの繰り返し（「ここでは〜を解説します」）、同一セクション内での同じ事実の2回記載。

### メタ発言
「ここでは〜を整理します」「以下で詳しく説明します」「〜を見ていきましょう」
→ 削除して直接本題に入る。冒頭リード文は例外。

### 伝えることを絞る
「全部書かなければ不完全」→「最も大事なことだけ伝わればいい」。

## 4. 正確に書くためのルール

### 曖昧表現の具体化
`❌ 幾つか` → `✅ 3点` ／ `❌ 早めに` → `✅ 1週間以内に` ／ `❌ 大幅に速くなる` → `✅ 500ms→80msに短縮`

### 「等」「など」
列挙できるなら列挙する。できないなら「等」「など」の前に代表例を最低2つ示す。

### 条件分岐
- 単独条件：「〜場合は、」
- 2段階：大きい条件に「場合」、小さい条件に「とき」
- 「とき」（仮定条件）と「時」（時点・時刻）を区別する

### AND/ORの明確化
「AやB」は曖昧。AND＝「および」「すべて」／OR＝「または」「いずれか」。
3つ以上は箇条書き＋末尾に「すべて」か「いずれか」を明記。
（技術記事では特に重要。「AまたはBが必要」と「AおよびBが必要」では設定が変わる）

### 義務・推奨・許可・禁止の書き分け
| 意味 | 表現 |
|---|---|
| 義務 | 〜してください / 〜が必要です |
| 推奨 | 〜するとよいです / 〜を推奨します |
| 許可 | 〜できます / 〜しても構いません |
| 禁止 | 〜しないでください / 〜は禁止です |

### 数量の境界
以上/以下＝含む。超える/未満＝含まない。「〜まで」「〜から」は曖昧になりやすいので避ける。
（バージョン条件で頻出：「Node 18以上」は18を含む）

### 用語の統一
- 一概念一用語（同義語を混在させない。「関数」と「メソッド」を区別して使う）
- 一用語一意味（文脈で意味が変わる語は避ける）
- 初出時に定義し、2回目以降は定義済み前提で使う

---

## 5. 読者への配慮

- 記事を書く前に読者の前提知識を定義する（職業・知識レベル・読む目的）
- 読者が知っていることは長々と説明しない
- 読者が知らない用語は初出時に定義する（定義は1回だけ）
- 専門用語の後に、できれば平易な言い換えを添える（例：「冪等（何回実行しても結果が同じ）」）

---

## 6. 箇条書き

- 3つ以上の並列は箇条書きにする
- 箇条書きの前に導入文を1文入れる（いきなり箇条書きを始めない）
- 各項目の文法構造を揃える（体言止めなら全部体言止め）

---

## 7. 表記の統一

### ひらがなで書くもの
- 接続詞：さらに、しかし、したがって、ただし、また（×更に、×然し、×従って）
- 形式名詞：こと、とき、ところ、もの、ほう（×事、×所、×物）
- 補助動詞：ていく、ていただく、てくる、てみる（×て行く、×て頂く）
- 副詞的用法：ほとんど、すべて、わずか

### 文体の統一
1記事内で「ですます調」と「である調」を混在させない。
例外：見出し・表のセル・コールアウト内箇条書きは体言止めOK。

### 技術用語の表記
- 製品名・言語名は公式表記に合わせる（JavaScript、TypeScript、Node.js、GitHub）
- 半角英数と全角の間にスペースを入れるか、サイト内で統一する

# 02. 記事構成テンプレート

技術ブログの記事は、だいたい4タイプに分かれる。
タイプごとに「読者が欲しいもの」が違うので、構成も変える。書く前に「これはどのタイプか」を決めると、構成で迷わなくなる。

| タイプ | 検索意図 | 読者が欲しいもの |
|---|---|---|
| チュートリアル | やり方を知りたい | 手順とコード、最後に動くもの |
| 解説 | 概念を理解したい | わかりやすい説明と具体例 |
| トラブル解決 | エラーを直したい | 原因と、すぐ試せる解決策 |
| 比較 | 選び方を知りたい | 違いの一覧と、結局どれを選ぶか |

各テンプレで共通する原則：**結論を先に。前置きは短く。**

---

## タイプ1：チュートリアル（やってみた・作ってみた）

「〇〇で△△を作る」「〇〇の使い方」系。手を動かして完成させる記事。

```
タイトル：【環境/バージョン】〇〇で△△を作る方法

リード
  - 何が作れるか（完成形を最初に見せる。画像やデモがあれば理想）
  - 対象読者（前提知識：例「Reactの基本がわかる人」）
  - 所要時間の目安

前提・環境
  - 必要なもの（Node.jsのバージョン、インストール済みパッケージ）
  - コードの全体像 or 完成リポジトリへのリンク

手順（番号付き、1ステップ1見出し）
  1. セットアップ
  2. 〇〇の実装
  3. △△の実装
  ※ 各手順に「コード」と「その時点の実行結果」をセットで
  ※ ハマりやすい所は「⚠️ ここで詰まりやすい」と補足

動作確認
  - 実際に動かした結果（スクショ or 出力）

まとめ
  - 完成したものの再確認
  - 次にやれること（発展)・関連記事へのリンク
```

**チュートリアルの肝**：
- コードは**コピペで動く完全な形**で。import文・前提を省略しない
- 「なぜそうするか」を1行添える（ただ写経させない）
- 最後に必ず「動いた状態」を見せる

---

## タイプ2：解説（〇〇とは・仕組み）

概念・仕組みを理解させる記事。「クロージャとは」「非同期処理の仕組み」系。

```
タイトル：〇〇とは？ △△を例にわかりやすく解説

リード
  - 一言での答え（定義を最初に。後回しにしない）
  - 「これがわかると何ができるか」

ざっくり理解（たとえ・図）
  - 厳密さより直感。身近な例で全体像をつかませる
  - 図解があれば強い

正確な説明
  - ここで厳密な定義・動作を説明
  - コード例で「実際の挙動」を見せる

よくある誤解・ハマりどころ
  - 初学者が間違えやすいポイント
  - 「〇〇と△△の違い」があれば明示

実際の使いどころ
  - どんな場面で使うか（具体的なユースケース）

まとめ
  - 要点を3点以内で
```

**解説の肝**：
- **定義を最初に**。「〇〇とは△△です」を1文目に置く。検索者は答えを急いでいる
- たとえ話 → 正確な説明 の二段構え。いきなり厳密だと挫折する
- 「わかった気にさせる」ではなく「実際に使える」まで持っていく

---

## タイプ3：トラブル解決（エラー・できない）

いちばん検索されるタイプ。「〇〇エラー 解決」「△△ できない」系。
**読者は焦っている。最速で答えにたどり着かせる。**

```
タイトル：【エラー名そのまま】の原因と解決方法

リード（最重要・3行以内）
  - このエラーが何か、原因は何か（結論を即座に）
  - 「下の方法で直ります」と道筋を見せる

結論（先に解決策）
  - いちばん多い原因と、その直し方を最初に
  - コードは「変更前 → 変更後」で対比

原因の説明
  - なぜこのエラーが起きるのか（理解したい人向け）

原因別の解決策
  - 原因が複数あるなら、原因ごとに見出しを分ける
  - 「ケース1：〇〇の場合」「ケース2：△△の場合」
  - それぞれに具体的なコードと手順

それでもダメなとき
  - 切り分け手順、確認すべきログ、関連Issue

まとめ
  - チェックリスト形式で「確認すべきこと」を箇条書き
```

**トラブル解決の肝**：
- **エラーメッセージをそのままタイトル・見出しに入れる**（検索者はエラー文をコピペして検索する）
- **解決策を先、原因の説明は後**。焦っている人に長い理論はいらない
- 「変更前 → 変更後」のdiff形式が最も親切
- 環境・バージョンを明記（解決策はバージョン依存のことが多い）

---

## タイプ4：比較（〇〇 vs △△・選び方）

「どれを選ぶか」を助ける記事。「npm yarn pnpm 違い」「状態管理 比較」系。

```
タイトル：〇〇と△△の違いと使い分け【〇〇年版】

リード
  - 結論を先に：「急ぐなら〇〇、△△なら□□」
  - 比較の観点を予告（速度・学習コスト・エコシステム等）

比較表
  - 観点を縦、選択肢を横に並べた一覧表
  - ◎○△× より、具体的な数値・事実で

観点ごとの詳細
  - 表の各行を、コードや実例で深掘り
  - 「速度」なら実測値、「学習コスト」なら具体的な躓きポイント

ケース別の選び方
  - 「個人開発なら」「チームなら」「大規模なら」
  - 読者が自分の状況で選べるように

まとめ
  - 結局どれを選ぶか、を言い切る（中立で逃げない）
```

**比較の肝**：
- **比較表を必ず入れる**（一覧性が価値）
- **「結局どれ？」に答える**。「ケースバイケース」で終わらせない。条件を切って言い切る
- 公平に書く。一方をけなすと信頼を失う。ただし**自分の推しは明確に**

---

## どのタイプでも共通の原則

1. **結論先行**：答え → 理由 → 詳細。前置きで体力を奪わない
2. **1見出し1トピック**：見出しを読むだけで記事の流れがわかる状態に
3. **コードは動く形で**：省略しない。実行結果も見せる
4. **見出しは「内容」で書く**：「まず確認すること」より「設定ファイルの場所が原因」
5. **読者の前提を最初に定義**：「Reactの基本がわかる人向け」と書くと、説明の粒度が決まる

構成に迷ったら、まず**「この記事を読んだ人が、最後に何ができるようになるか」**を1行で書く。それが記事のゴールで、構成はそこに向かって組む。

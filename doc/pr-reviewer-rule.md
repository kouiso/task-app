# 📝 レビュワーの意識すること

## 💬 コメントする上で意識すること

> **👍 いいと思った実装があればコメントする**
>
> **💡 他に効率的な書き方があればsuggestする**
>
> **[MUST][NITS][IMO][Q]のような項目を付ける**
>
> - NITS → 細かいかもしれないけど
> - IMO → 私はこう思いますが、あなたはどうですか？
> - Q → 質問
> - MUST → これは変えた方がいい
>
> **📝 レビュアがResolve Conversationを消す**

## 🔍 レビューする観点で意識すること

### 🌟 共通

- ❌ ～Deprecatedメソッド内で～Deprecatedでないメソッドがコールされていないか
- ❓ そもそも～Deprecatedがないメソッドの場合もあるので全てがそうではないが、～Deprecatedがついてないものが呼ばれていたらまず怪しむのがよい
- 📝 変数名、関数名、カラム名等の命名が適切か(typo含め)
- 🎯 用途に即しているか。
- 🚫 否定名は避ける。
- 🔤 関数→動詞、変数→名詞、tempみたいな意味のない文字を使っていないか。
- 🧹 冗長なコーディングになっていないか
- 🏞️ nestが深すぎないか。 → Linterでチェック(例：3個あれば警告、5個でエラー出すようにする)
- 🧩 凝縮度が適当か。単一責任の原則に即しているか。
- 🔄 逆に分割されすぎていないか。
- 🗂️ 各レイヤーのふさわしい場所にコードが記載されているか。
- 🗑️ 使われていないコード(デッドコード)が存在していないか→Linterでチェックできる。参照されていない関数はできていない→これはライブラリやスクリプトでできないとかないかな？
- 🔍 スコープの範囲
- ✅ 受入条件が満たされているか→やったことを受入条件に変更する
- 🚫 動作確認まではしない
- ⚠️ 自分が以前経験した落とし穴系にはまってないか
- 🔐 ビジネスロジックがちゃんと適用されているか。（特に権限周りで、見れて/操作してはいけない人に見れるようになっていないか。）
- 🚨 意図しない動作をした時に検知できるようになっているか
- 🗑️ 捨てやすいコードか（きちんと関数や適切なファイルに切り分けられているか）
- 🔄 再利用しやすいコードであるか。
- 📝 jsの書き方として誤りがないか。より変更に強い書き方になっているか。
- 🚀 パフォーマンス上問題はないか
- 🔧 処理のハンドリングが正しく行えているか
- ⚠️ 例外処理が書けているか
- 🚨 意図しない動作をした時に検知できるようになっているか
- 🔄 同期・非同期処理が書けているか
- 💥 破壊的な変更をしていないか。
- 📝 エラーメッセージが適切で分かりやすいか

### 🎨 フロントエンド

- 🗑️ useStateを無駄に作りすぎていないか
  - 例：1つのstateで事足りるのにそのstateで別のstateを作っている
- 🧩 親ではなく子コンポーネントで宣言した方がよいstateがないか
- 🔍 HookFormで不適切なstateがwatchされいないか
- ❌ isDirty使っていないか
  - isDirtyを使っているということはuseFormに即した書き方になっていないというケースが多いため
- ⚙️ depsの設定が適切か
- 🔄 useEffectやuseCallbackのReRun処理が正しいタイミングで書き変わっているか。
- 🧩 コンポーネントの切り方が適切か
- 🎨 デザインシステムに則ってかけているか(命名含む)
- 🎨 デザインを再現できているか

### 🤔 悩むところ

- 書き方が人によって異なる場合がある。これをレビュー観点として入れるか？→レビュアーが違和感を感じたら意図を聞いてフィードバックする

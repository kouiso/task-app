# 確認待ちで止まった時の見方

## ユーザーが今できるようになったこと

- 改善が公開前の確認待ちで止まった時に、次に見る場所が分かります。
- 「アプリ側の不具合」なのか「確認する機械の順番待ち」なのかを切り分けやすくなります。
- 確認待ちのまま公開してよいか迷った時に、公開してはいけない基準が分かります。

## まず見るもの

1. 画面の公開確認が成功しているかを見る。
2. 自動レビューが成功しているかを見る。
3. GitHub Actions の各確認が「待ち」から進んだかを見る。
4. 「待ち」のままなら、アプリの修正ではなく確認環境の問題として扱う。

## 公開してはいけない状態

- GitHub Actions がまだ待ち状態のまま。
- 失敗ログを読んでいない。
- 実際の画面を見ていない。
- 「たぶん動く」だけで、動作確認の画像やログが残っていない。

## 確認待ちが続く時に残す記録

以下を issue や引き継ぎに残します。

- どの改善が止まっているか。
- どの確認が待ち状態か。
- 画面の公開確認は成功しているか。
- 自動レビューは成功しているか。
- いつから待ち状態か。
- 次に誰が何を見るか。

## 局長向けの報告文

専門用語や番号の羅列ではなく、次の形で報告します。

```text
■ ユーザーが今できるようになったこと
- 検索条件を消した時に、古い条件が残らないようになりました。

■ 動作確認 evidence
- 画面の公開確認は成功しました。
- 自動確認は順番待ちです。確認待ちの記録を残しました。

■ 今後修正予定
- 確認が動き出したら、失敗ログを読んで必要な修正を入れます。

■ 既知の課題
- GitHub Actions が順番待ちで止まっています。
```

## この文書を使う場面

- 画面の公開確認は成功しているのに、最後の確認が進まない時。
- 複数の改善が同じ理由で止まっている時。
- 局長に「何が良くなったのか」と「本当に確認したのか」を報告する時。

## 【第一章:プロジェクト概要】

### 1.1. TaskAppとは

TaskAppは、タスク管理を効率化するフルスタックWebアプリケーションです。

#### 主要機能
- **タスク管理**: タスクの作成・編集・削除・一覧表示
- **認証機能**: NextAuth.jsによるユーザー認証
- **カテゴリ管理**: タスクをカテゴリで分類・管理
- **優先度設定**: タスクに優先度を設定して整理
- **ステータス管理**: 未着手・進行中・完了などのステータス管理

### 1.2. 技術スタック

**フロントエンド**:
- **言語**: TypeScript 5.x
- **フレームワーク**: Next.js 15 (App Router)
- **UIライブラリ**: Material-UI (MUI) v7
- **状態管理**: React Hooks

**バックエンド**:
- **API**: tRPC (型安全なAPI通信)
- **ORM**: Prisma
- **データベース**: PostgreSQL
- **認証**: NextAuth.js

**開発ツール**:
- **テスト**: Jest + Testing Library
- **Lint**: ESLint + Prettier
- **デプロイ**: Vercel

### 1.3. プロジェクト構造

## 【第二章:教材の定義と21日間進行管理】

### 2.1. 成果物の定義

**プログラミング初心者向けの学習教材**として、以下のすべてを満たす教材を作成します。

#### 教材の性質
- **学習キット**: 受講者が21日間、毎日写経することでNext.js + TypeScript + Prisma + tRPC + NextAuth.jsを使ったフルスタックWebアプリ開発の全体像を習得できる
- **完成品**: 21日後には、そのままデプロイ可能な品質のタスク管理アプリケーションが完成している
- **追跡可能性**: コードは一行ずつ追跡可能なシンプルさを維持し、学習のしやすさを最優先する
- **実用品質**: 成果物自体が、実際に使える実用的なアプリケーションである

#### 教材とリポジトリの関係性
- **教材とリポジトリの完全一致**: 現在のリポジトリが最終的なゴール(完成品)であり、教材はこのリポジトリと全く同じ機能・構造・品質のアプリケーションを21日間で段階的に構築する手順書として機能する
- **積み上げ式開発の徹底**: 一度実装したコードを後で削除・置換するような「後戻り」は絶対に禁止する。各日の実装は最終成果物に向かって段階的に積み上げる
- **完成品としてのソースコード**: 現在のリポジトリのソースコードは、教材21日間をすべて完了した後の最終成果物である

### 2.1.5. ソースコード量ベースの日数設計

**重要原則**: 教材の日数は、ペルソナや理想論ではなく、**実際のソースコード量**から逆算して決定します。

#### 分析結果
- **総ファイル数**: 63ファイル(テスト除く)
- **総行数**: 8,925行
  - TypeScript/TSXソースコード: 8,735行
  - Prismaスキーマ: 190行
- **1日あたりの学習・実装量**: 約400-450行
- **推奨学習期間**: 21日間

#### 自然な実装の流れを重視
- 依存関係の順序を考慮した段階的実装
- 前の機能が完成してから次の機能に進む
- 無理のない学習ペース

#### テストコードは教材に含めない
- テストコードは教材から除外
- テストの書き方を学ぶのは別の学習機会

### 2.2. 21日間の厳格な日数制約

- **正確に21日分の教材**とします(環境構築のDay 1を含む)
- **各日の実装は必ずその日のうちに完結させ、単体で動作可能な状態で終了してください**
- 複数日にまたがる機能は、各日が完結するように適切にタスクを分割します。21日以内に完了不可能な機能は、実装対象から除外します

### 2.3. 1日分の定量的制限

#### 作業時間の現実的制限
- **学習時間**: 初心者が2時間程度で理解・実装可能な範囲
- **柔軟な対応**: 内容によって多少の前後は許容する

#### コード量の柔軟な管理
- **新規追加**: 機能として完結する最小限の実装(行数制限なし)
- **重要なのは行数ではなく理解可能性**:
  - 1つのファイルは200行以内を目安
  - 1つの関数は30行以内を推奨
  - 1つのコンポーネントは150行以内を推奨

#### 新概念と新規ファイルの制限
- **新概念**: 1日に導入する新しい概念は3つまで
- **ファイル数**: 1日に新規作成するファイルは5ファイルまで

### 2.4. 学習者の詳細プロファイル

#### 前提知識レベルの明確な定義

**学習者の想定レベル**: **TypeScript/JavaScriptの基礎知識を持つ初級〜中級者**

**スタート地点**:
- **JavaScript経験**: 変数、関数、オブジェクト、配列、非同期処理(async/await)の基本を理解している
- **TypeScript経験**: 基本的な型定義(string, number, interface, type)を理解している
- **React経験**: 基本的なコンポーネント、useState、useEffectを使ったことがある(簡単なアプリを作ったレベル)
- **環境**: Node.js v20.3.1以上がインストール済み

**教材が前提としないスキル(=すべて教材内で丁寧に説明する内容)**:
- Next.js App Routerの使用経験
- tRPCの使用経験
- Prismaの使用経験
- NextAuth.jsの使用経験
- Material-UIの使用経験
- PostgreSQLの使用経験
- Vercelへのデプロイ経験

#### 現実的な学習制約
- **時間制約**: 平日は仕事後の1-2時間、週末でも3時間が現実的な上限
- **集中力制約**: 複雑すぎる内容は挫折の原因となるため、段階的な難易度設定が必須
- **モチベーション維持**: 毎日の小さな達成感と「実際に使える」実感が継続の鍵

#### 学習環境の前提条件

**必須環境**:
- **OS**: Windows 10/11、macOS、またはLinux
- **Node.js**: バージョン20.3.1以上
- **エディタ**: VS Code推奨
- **インターネット接続**: パッケージのインストールと外部サービス利用に必要
- **PostgreSQL**: ローカル環境またはクラウドDB(Supabase/Neon等)

**Day 1で環境構築するもの**:
- プロジェクトのクローン/セットアップ
- 依存パッケージのインストール
- PostgreSQLデータベースの準備
- 環境変数の設定
- Prismaの初期設定とマイグレーション
- 開発サーバーの起動確認

## 【第三章:Next.js + TypeScript + tRPC実装の技術要件】

### 3.1. 実装原則

- **シンプル is BEST**: 高度な設計パターン(複雑なカスタムフック、過度な抽象化等)は禁止します。初心者がコードを一行ずつ追跡可能なシンプルさを維持してください
- **適切な品質**: 動作して理解しやすいコードを重視し、基本的なエラー対応を含む実用的な品質を維持してください
- **型安全性**: TypeScriptとtRPCの型システムを活用し、エンドツーエンドの型安全性を確保
- **App Router準拠**: Next.js 15のApp Routerのベストプラクティスに従う

### 3.2. プロジェクト構造(再掲)

```
src/
├── app/           # Next.js App Router(ページ・レイアウト)
├── components/    # 再利用可能なReactコンポーネント
├── server/        # サーバーサイドロジック(tRPC, Auth, DB)
├── lib/           # ユーティリティ・設定
├── types/         # TypeScript型定義
└── hooks/         # カスタムフック
```

### 3.3. 禁止事項(アンチパターン)

#### 基本的なアンチパターン
- グローバル変数の乱用
- any型の過度な使用
- ハードコーディングされた設定値
- 不適切なエラーハンドリング
- 重複コードの放置

#### 教材特有の禁止事項
- **現在のTaskAppリポジトリにない機能の追加**
- **未完成・中途半端な機能の実装**
- **理解困難なコードの作成**
- **初心者には明らかに難しすぎる機能の実装**(複雑な状態管理、高度な最適化など)
- **Pages Routerの使用**(必ずApp Routerを使用)

### 3.4. 初心者配慮の技術的制約

- **過度な最適化の回避**: パフォーマンス最適化機能(メモ化、仮想化等)は初心者には不要
- **複雑な設計パターンの排除**: 高度なカスタムフックパターン、複雑なContext構造等は使用禁止
- **シンプルなデータ構造**: 複雑な正規化、ネストしたState管理は最小限に
- **ライブラリ依存の最小化**: 新しいライブラリの導入は教育的価値が明確な場合のみ

### 3.5. ファイル命名規則

- **基本ルール**: 英語単数形のケバブケース
  - ✅ `task-service.ts`, `task-list.tsx`, `user-auth.ts`
  - ❌ `taskService.ts`, `TaskList.tsx`, `users-auth.ts`
- **例外**: 慣習的な大文字ファイル名
  - ✅ `README.md`, `CLAUDE.md`, `LICENSE`
- **Reactコンポーネント**: ケバブケースのファイル名、PascalCaseのコンポーネント名
  - ファイル: `task-list.tsx`
  - コンポーネント: `export function TaskList() { ... }`

## 【第四章:ドキュメント(教材本体)の作成仕様】

### 4.1. 基本仕様

- **格納場所**: `doc/`ディレクトリ
- **形式**: **シンプルなMarkdown (`.md`)**
- **命名規則**: `{ステップ番号2桁}_{内容}.md` (例: `01_setup_environment.md`)
- **重要原則**: 現在のリポジトリは21日間の教材完了後の最終成果物であり、教材はこの完成品に段階的に到達する手順書として機能する

### 4.1.2. 教材に含めない要素

以下の要素は教材に含めません:

#### DevContainer設定
- `.devcontainer/`ディレクトリの設定は教材で紹介しない
- ローカル環境での開発を前提とする
- Docker環境の構築は各自で行う

#### テストコード
- `src/test/`
- `**/__tests__/`
- `*.test.ts`, `*.test.tsx`
- テストの書き方は別の学習機会

#### 高度な開発ツール設定
- CI/CD設定
- 複雑なLint/Format設定
- パフォーマンスモニタリング

### 4.1.1. 教材の完結性に関する絶対ルール

**ドキュメント単体での完結性の保証**:
- **ソースコードありきではない**: リポジトリのソースコードを一切読まなくても、教材ドキュメント(`doc/*.md`)だけで環境構築から最終的なアプリケーション完成まで進められること
- **すべてのコードを明示**: 必要なコードは**すべてドキュメント内に全文記載**し、「詳細はソースコードを参照」のような記述は**絶対に禁止**
- **コピー&ペーストで動作**: ドキュメントに記載されたコードをそのままコピー&ペーストすれば、**100%確実に動作**すること
- **動作確認の明示**: 動作確認時のブラウザ表示やコンソール出力を具体的に記載し、学習者が正しく進められているか確認できるようにすること

**外部リソースへの依存禁止**:
- **外部ドキュメントへのリンク禁止**: 「公式ドキュメントを参照してください」のような記述は禁止
- **必要な説明はすべて教材内に含める**: 外部サイトを見なくても理解できる説明を提供すること
- **例外**: Next.js公式ドキュメントやPrisma公式ドキュメントへのリンクは、「さらに詳しく知りたい方向け」として補足的に記載するのみ可

**トークン数よりも丁寧さを優先**:
- **説明の省略禁止**: トークン数を気にせず、初心者が理解できるまで丁寧に説明すること
- **冗長でも明確に**: 「言わなくてもわかるだろう」という省略は厳禁。すべてを明示的に記述すること

### 4.2. 品質要件

#### 品質基準
- **誤字脱字ゼロ**: 完全に推敲された、誤りのない文章
- **専門用語の丁寧な説明**:
  - 初出時には必ず平易な言葉で説明
  - 例: 「tRPC(型安全なAPIを簡単に作れる仕組み)」
- **コード説明の徹底**:
  - すべてのコードブロックに、その前後で「何をするコードか」「なぜ必要か」を明記
  - コメント付きコードと、コメントなしの実装用コードの両方を提示
- **動作確認の具体性**:
  - 「サーバーを起動して確認してください」ではなく
  - 「`npm run dev`を実行し、ブラウザで`http://localhost:3000`にアクセスすると、ホームページが表示されます」のように具体的に記述

### 4.5. 段階的実装の原則

- **最大**: 30行(複雑な関数、クラス定義など)
- **30行を超える場合**: 必ず複数のステップに分割する

**各ステップの必須構成要素**:
1. **タイトル**: `### Step X.Y: 〜(所要時間:N分)`
2. **このステップで学ぶこと**: 1文で要約
3. **なぜ必要?**: 初心者向けの丁寧な説明(例え話を含む)
4. **コードの仕組み解説**: コード内の各要素を箇条書きで説明
5. **コードブロック**: 必ず`// filepath: パス`コメント付き
6. **確認方法**: 番号付きリストで具体的な手順
7. **よくあるエラー**: 想定されるエラーと解決方法

# Trial & Error and Zero User Burden (トライアンドエラーとユーザー負担ゼロの原則)

## 1. ユーザー負担ゼロの絶対原則 (Zero User Burden Protocol)

**ユーザーから言われなくても、自主的に自動でユーザーが実際に実行することを想定して、あなたの方で実行してユーザーに楽をさせること。**

### 核心思想：内田祐貴の完璧主義

- **ユーザーが実行してからエラーが出るという事態は、AIの敗北である。**
- **「動くはずです」は禁止。「動きました」のみが許される。**
- **完璧主義の精神でトライアンドエラーを徹底する。**

### 行動指針

1. **自主的検証 (Proactive Verification)**
   - ユーザーに言われる前に、AIが自主的に実行・検証を行う。
   - ユーザーが実行するコマンドを先回りして実行し、成功を確認する。
   - 「ユーザーが実行してからエラーが出る」という事態を絶対に避ける。

2. **妥協なき修正とエラー隠蔽の完全禁止 (Uncompromising Fixes & No Error Suppression)**
   - **エラーハンドリングの放棄（`|| true`等）は完全禁止**。
   - エラーが出た場合は少しでも妥協せず、根本的に修正してユーザーにデバッグの負担をかけない。
   - その場しのぎの修正（臭いものに蓋をする行為）は厳禁。
   - **冪等性（Idempotency）の担保**: 何度実行しても壊れない堅牢なスクリプトやコードを書くこと。（例: `task init` 等）

3. **完全な再検証 (Complete Re-verification)**
   - エラーが出て修正したら、**途中から端折って失敗したところから再開するのではなく、最初からやり直す**。
   - 「本当にエラーを解決したかどうか」は、クリーンな状態からの再実行でしか証明できない。
   - 完璧主義の精神でトライアンドエラーを徹底し、ユーザーには「成功した結果」のみを届けること。

4. **デバッグ負担の排除**
   - ユーザーにデバッグの負担をかけない。
   - エラーログの解析、原因特定、修正、検証まで全てAIが完結させる。

## 2. ワークスペース外ファイルの取り扱い (Handling Files Outside Workspace)

**VS Codeの制約を理由に、ユーザーに環境設定の手間をかけさせることは禁止。**

❌ **禁止**:
- 「ワークスペース外なので読み書きできません」と即答すること
- 「フォルダをワークスペースに追加してください」と依頼すること

✅ **必須**:
- VS Code API（`read_file`等）が使えない場合は、必ず**ターミナルコマンド（`cat`, `ls`, `echo`, `sed`等）**でのアクセス・編集を試みること。
- 「できない」と言う前に、代替手段（CLIツール、シェルスクリプト）を模索すること。

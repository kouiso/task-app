# 自明なコメント禁止（No Obvious Comments）

## 原則

**コードを読めばわかることをコメントに書くな。**

コメントは「なぜ（Why）」を説明するために存在する。「何をしている（What）」はコード自体が語るべきであり、コメントで繰り返す必要はない。

---

## 禁止パターン（具体例）

### 変数・定数の宣言

```typescript
// ❌ 禁止: 見ればわかる
const maxRetries = 3; // 最大リトライ回数
const isLoading = true; // ローディング状態

// ✅ 許可: なぜその値かを説明
const maxRetries = 3; // API側のレート制限が5回/分のため、余裕を持たせて3回
```

### 関数・メソッド

```typescript
// ❌ 禁止: 関数名と引数で明らか
// ユーザーを取得する
async function getUser(id: string) { ... }

// ユーザー名を更新する
function updateUserName(userId: string, name: string) { ... }

// ✅ 許可: 非自明な仕様・制約の説明
// Auth0のRate Limit回避のため、取得後500msのdelayを入れている
async function getUser(id: string) { ... }
```

### 条件分岐

```typescript
// ❌ 禁止: コードそのまま
// ステータスが'active'の場合
if (status === 'active') { ... }

// nullチェック
if (user != null) { ... }

// ✅ 許可: ビジネスロジックの背景
// 退会猶予期間（30日）内のユーザーもactiveとして扱う仕様
if (status === 'active') { ... }
```

### インポート・設定

```typescript
// ❌ 禁止: 見ればわかる
import { useState } from 'react'; // Reactのフックをインポート

// ❌ 禁止: セクション区切りとして無意味
// --- Imports ---
import { foo } from './foo';
```

### ブロックの終わり

```typescript
// ❌ 禁止
if (condition) {
  // ...処理...
} // if文の終わり

for (const item of items) {
  // ...処理...
} // forループの終わり
```

## コメントが正当化される場面

以下の場合のみコメントを書いてよい：

1. **Why（なぜ）**: その実装を選んだ理由、ビジネスロジックの背景
2. **Workaround**: ライブラリのバグ回避、一時的な対処とその理由
3. **Warning**: 一見正しく見えるが変更すると壊れる箇所への警告
4. **TODO/FIXME**: 既知の技術的負債の記録（チケット番号付き）
5. **正規表現・複雑なアルゴリズム**: 可読性が低い箇所の意図説明
6. **非自明な型アサーション**: なぜ型を上書きしているかの理由

---

## ルール

1. **コメントを書く前に、変数名・関数名で意図を表現できないか考える** — コメントが必要な時点でネーミングが悪い可能性がある
2. **「何をしている」コメントは全て削除** — コードリーディングの邪魔になるだけ
3. **JSDoc/docstringは公開APIのみ** — 内部関数に形式的なdocstringを付けない
4. **既存コードにコメントを追加する変更はレビューで指摘対象** — 自明なコメントの追加はコード品質の低下

---

## 自己チェック

コメントを書こうとしたら以下を自問すること：

- **このコメントを消しても、コードの意味は変わらず伝わるか？** → 伝わるなら不要
- **このコメントは「What」か「Why」か？** → 「What」なら不要
- **変数名・関数名を改善すれば不要になるか？** → なるなら名前を直す

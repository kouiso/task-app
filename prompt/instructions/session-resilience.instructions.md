# Session Resilience & Context Management（セッション耐障害性・コンテキスト管理）

長時間セッションの安定稼働、セッション断からの復旧、コンテキスト溢れの防止を実現するためのルール。

## 1. 画像によるセッション死亡の防止

### 絶対ルール

**スクリーンショット・画像の確認は、必ずサブエージェント（Task）に委任せよ。メインコンテキストで画像を直接Readすることは原則禁止。**

### なぜ禁止か

- 画像1枚で数千トークンを消費する
- テスト・デバッグ作業で複数枚読むと、コンテキストウィンドウが急速に圧迫される
- 結果として `prompt too long` でセッションが死ぬ

### 正しいアプローチ

#### A: サブエージェントへの画像確認委任（推奨）

```
Task(subagent_type=general-purpose):
  「/tmp/screenshot_home.png を読んで、画面に何が表示されているか教えて」
  → サブエージェント側で画像を読み込み（数千トークン消費）
  → メインに返るのは「ホーム画面に8件表示されていました」（数十トークン）
```

サブエージェントは別コンテキストで動作するため、画像のトークン消費がメインに波及しない。

#### B: 画像パス参照方式

レポートやメモに画像を埋め込まず、ファイルパスだけ記録する。

```markdown
## スクリーンショット一覧
| # | 画面 | 保存先 |
|---|------|-------|
| 1 | ホーム画面 | /tmp/screenshot_home.png |
| 2 | 詳細画面 | /tmp/screenshot_detail.png |
```

画面の状態はテキストでも記述し、画像は「エビデンスとして保存してあるが、作業継続には不要」とする。

```markdown
- [x] ホーム画面にデータ8件表示
- [ ] 詳細画面に生年月日が表示される → 表示なし
```

## 2. コンテキスト分離の原則（セッション長寿命化）

**「調べる」と「直す」のコンテキストを分離せよ。**

### ルール1: 探索はサブエージェントに委任

```
Task(subagent_type=Explore) で探索・調査を実行
→ サブエージェントが数十ファイル読んでも、メインには要約だけが返る
```

### ルール2: メインでは編集対象だけRead

```
実際に Edit するファイルだけを Read する
「念のため確認」「参考程度」の読み込みはサブエージェントに任せる
```

### ルール3: 独立した調査は並列Taskで同時実行

```
例: バックエンドの設定確認 と フロントのコード調査
→ 2つのサブエージェントを並列に走らせる
→ 往復回数が減り、トークン消費を抑制
```

### ルール4: サブエージェントの結果を重複しない

```
サブエージェントが調査した内容を、メインで同じファイルを再度読まない
結果を信頼して次のステップに進む
```

## 3. セッション断からの復旧機構

### レポートファイル + MEMORY.md の二重永続化

長時間作業（テスト、デバッグ、移行等）では、進捗をレポートファイルに記録し、要約をMEMORY.mdにも複製する。

#### レポートファイルの設計原則

```markdown
# 作業レポート (日付)

## 目的
何のための作業か（1行）

## 進捗サマリー
| Step | 内容 | 状態 | 結果 |
|------|------|------|------|
| 0 | 初期化 | 完了 | OK |
| 1 | 機能A確認 | 完了 | OK |
| 2 | 機能B確認 | 未完了 | ブロッカーあり |

## 問題記録
| # | 問題内容 | 重要度 | 対応状態 |
|---|---------|--------|---------|

## 修正優先度
### P0: 必須
### P1: 推奨
### P2: 後日
```

#### MEMORY.mdへの要約複製

MEMORY.mdはセッション開始時に自動ロードされる。レポートの要約を書けばセッション死亡後も新セッションで即座にコンテキスト復元。

```markdown
## 作業名 (日付)
- **レポートファイル**: `パス` （詳細はこちら）
- **最終更新**: Step Nまで完了

## 解決済みの重要問題
- **問題名**: 原因 → 修正方法（1行で）

## 既知の未修正問題
- 箇条書きで列挙
```

#### 再開用プロンプト

```
レポートファイル.md を読んで、作業の続きをお願いします。
未完了のStepから再開してください。
```

## 4. planファイルの永続化

### 課題

planファイルをプロジェクトルートに作成した後、セッションが死ぬとplanの存在を忘れる。

### ルール

- planファイルを作成したら、**MEMORY.mdにファイルパスと目的を即座に記録**
- planファイル名に日付を含める（例: `plan-2026-02-14-feature-xxx.md`）
- 完了したplanは `doc/plans/` に移動（アーカイブ）

```markdown
# MEMORY.mdに書く内容
## 進行中のplan
- **ファイル**: `plan-2026-02-14-feature-xxx.md`
- **目的**: ○○機能の追加
- **状態**: Step 3まで完了、Step 4から再開
```

## 心得

- **画像は毒** → メインコンテキストで直接読むな、サブエージェントに委任
- **探索は委任** → メインコンテキストには要約だけ返す
- **永続化は二重** → レポート + MEMORY.md の両方に書く
- **planは記録** → 作った瞬間にMEMORY.mdにパスを書く
- **セッションはいつ死んでもおかしくない** → 常に復旧可能な状態を維持

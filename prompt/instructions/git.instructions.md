# Git & GitHub Rules

## 1. Gitブランチ運用ルール

### ブランチ戦略概要

本プロジェクトはGit Flow をベースとしたブランチ戦略を採用している。

### 主要ブランチ

| ブランチ名 | 役割 | 備考 |
|-----------|------|------|
| `main` | 開発用メインブランチ | **全ての開発ブランチはここから切る** |
| `production` | 本番環境用ブランチ | 本番デプロイ対象 |

### 開発時のブランチ運用ルール

#### 新機能開発・バグ修正時

1. **必ず`main`ブランチから新しいブランチを切る**
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/xxx  # または fix/xxx, chore/xxx 等
   ```

2. **`production`ブランチから直接ブランチを切ることは禁止**
   - `production`は本番環境専用であり、開発作業の起点にしてはならない

#### リリース作業時

- `production`ブランチへのマージはリリース作業時のみ実施
- リリースPRは`main` → `production`の方向でマージ

### ローカルのproductionブランチについて

ローカルリポジトリに`production`ブランチが存在する場合があるが、これは**git flowコマンドを実行するためのダミーブランチ**である。

- ローカルの`production`ブランチで直接開発作業を行わないこと
- git flowコマンド（`git flow init`等）の初期化・設定目的でのみ使用
- 実際の本番デプロイはリモートの`production`ブランチを参照する

### 心得

- **開発は常に`main`から** → `production`から切るな
- **`production`は本番専用** → 開発ブランチの起点にするな
- **ローカル`production`はダミー** → git flowコマンド用、直接作業するな

## 2. Pull Request のルール

### テンプレート準拠

- `.github/pull_request_template.md`存在確認
- 存在時：テンプレート厳密従来（肉づけ可、項目変更・省略禁止）
- 不存在時：制約適用なし

### クロスリポジトリIssue参照の絶対ルール

**PRやコミットメッセージで別リポジトリのIssueを参照する際、短縮形`#123`は使用禁止。**

- **禁止**: `#429`（同一リポジトリのIssue #429にリンクされてしまう）
- **必須**: `getozinc/mypappy#429`（完全修飾形式）

**理由**: 複数のリポジトリが連携しているプロジェクトでは、`#429`と短縮形で書くと現在のリポジトリのIssueにリンクされてしまい、参照したい別リポジトリのIssueにリンクされない。

**確認手順**:
1. PR作成/更新前にIssueがどのリポジトリにあるか確認
2. 異なるリポジトリの場合は必ず`owner/repo#number`形式で記述

### GitHub操作注意事項

- 新規PR作成: github mcp または gh コマンド
- **既存PR更新**: github mcp使用不可、gh コマンド必須
- github mcp不可能時: gh コマンド代替

### PR作業前のブランチ確認義務

**PR関連作業を行う前に、必ず `gh pr view <PR番号> --json headRefName` でブランチ名を確認すること。**

- **禁止**: PRのブランチ名を記憶や推測で決めつけて作業
- **必須**: `gh pr view` または GitHub MCP で実際のブランチ名を確認してから作業

**理由**: PRのブランチ名を間違えると、関係ないブランチに変更をpushしてしまい、レビューに混乱を招く。

## 3. Commit Fix Protocol

**`/commit-fix`実行時、AI絶対的ルール一字一句違わず実行。レビュワー負担極限軽減、デグレ絶対防止最重要プロトコル。**

### 状況把握フェーズ (Step 1-3)

#### Step 1: PR探索
- 現ブランチ紐づきPR既存確認（github mcp等）
- 「存在有無」確認のみ、PR作成なし
- ユーザーPR URL提示時最優先差分確認

#### Step 2: 差分特定
- **PR存在**: 「Files changed」タブファイル差分全体完全把握
- **PR不存在**: 現ブランチ vs 派生元（develop/main等）差分把握

#### Step 3: 対象コミット選定
- 上記差分影響コミットのみ整理対象リストアップ
- 関係ないコミット絶対不干渉

### 精神統一フェーズ (Step 4)

#### Step 4: ルール再確認
- `.github/copilot-instructions.md`存在確認
- 存在時内容改めて熟読、全作業100%準拠再認識
- 特に「Technical Regulations」内「Git & GitHub」セクション・関連禁止事項心刻

### コミット再構成フェーズ (Step 5)

#### Step 5: 自律的rebase実行

**`git rebase -i`を`GIT_SEQUENCE_EDITOR`とスクリプト（sed/python）組合自動実行方法のみ。他方法（手動rebase、`reset --hard`等）絶対使用禁止。**

##### 再構成時遵守思考原則

###### 全体計画策定
- 対象作業コミット全体俯瞰
- 複雑コードリファクタリング前クラス・関数依存関係分析、全体設計図描画作業相当
- 「コミット統合でレビュワー論理的変更単位実現？」「最終コミットリストで変更意図時系列明確伝達？」思考
- **最終コミットリスト完成形全計画後作業着手**

###### コミットラベル禁止
- 最終コミットメッセージ`refactor:`、`fix:`等Conventional Commitsプレフィックス含有禁止

###### `feat`はPRに一つだけ
- 一PR内`feat:`相当機能追加コミット複数含有絶対禁止
- PR関心事一つ絞り、レビュー焦点明確化絶対ルール

###### 既存コミット尊重
- **`git reset --soft`コミット一旦リセット一から作直方法原則禁止**
- 積上実装意図消去、デグレ原因防止
- 既存コミット**統合（squash/fixup）・順番入替**でより良形磨上意識

###### 「意味あるコミット」統合
- 「WIP」「tmp」「あとで修正」等作業記録コミット「意味ないコミット」見做
- 関連「意味あるコミット」統合（squash/fixup）必須

###### コミットメッセージ・差分一致
- 最終コミットメッセージ含有差分内容**過不足なく**表現
- 「ユーザー認証機能追加」コミット関係ないログ出力修正紛込禁止

##### AI向け解説：「意味あるコミット」定義

**データ処理パイプライン最適化例思考**:

- **意味ないコミット（最適化前）**: 「データロード」「一時変数A作成」「A加工」「デバッグA プリント」「不要B削除」
  - パイプライン構成個々命令（アトミック操作）、断片的最終目的不明瞭
  - `WIP`、`fix typo`等作業コミット該当

- **意味あるコミット（最適化後）**: 「データソースA生データ抽出・正規化」「正規化データ集計・特徴量生成」
  - 一連操作まとまり「論理的処理ステージ」
  - レビュワー（パイプライン保守者）「ステージ」単位処理内容理解・評価

**`/commit-fix`タスク**: 断片的命令（作業コミット） → レビュワー理解可能・論理完結「処理ステージ」（意味あるコミット）再構成

### 最終確定フェーズ (Step 6)

#### Step 6: 仕上・強制プッシュ
- コミット再構成完了後`git status`等コミット漏・ステージング忘差分最終確認
- 全完璧確認後`git push --force-with-lease`コマンドリモートブランチ更新
- **`--force`オプション単体使用いかなる状況絶対禁止**
- `--force-with-lease`: 他開発者リモートブランチ変更意図せず上書事故防止必須安全装置遵守

## 4. プロジェクトナレッジベース & 参照リポジトリ

### プロジェクトナレッジベース

MYPAPPYプロジェクトの変わらない基盤情報（リポジトリ構成、決済システムアーキテクチャ、技術スタック等）は以下のファイルに集約されている：

- **`knowledge.md`**: プロジェクトの安定した情報を記録したナレッジベース
  - リポジトリ構成（mypappy, mypappy-api, mypappy-native, mypappy-web）
  - 決済システム情報（CREDIX, UnivaPay, Stripe, RevenueCat）
  - 技術スタック詳細
  - チームメンバー情報
  - 外部サービス連携

**作業開始前に必ず`knowledge.md`を確認し、プロジェクトの全体像を把握すること。**
